### Why the difference?

**1. CVE-2026-5450** (libc6, Critical)
- **Found by:** Grype
- **Missed by:** Trivy
- **Likely reason:** The discrepancy is likely due to differences in the vulnerability databases used. Grype uses the Anchore vulnerability database, which includes data from NVD and GitHub advisories, while Trivy maintains its own database. In this case, Trivy could have classified the CVE as outside the scope of the Debian package (e.g., "will not be fixed") or simply not included it in the database during scanning.

**2. CVE-2024-37890** (ws, High)
- **Found by:** Trivy
- **Missed by:** Grype
- **Likely reason:** The most likely explanation is how the two tools locate packages. Trivy directly inspects the container image and therefore discovered the `ws` package on the filesystem. Grype, on the other hand, operates on the SBOM generated by Syft. If the SBOM did not correctly represent the `ws` package, Grype had no way of associating a vulnerability with it. This case illustrates how dependent Grype's results are on the completeness of the SBOM.

### When would you pick each?

The decoupled Syft+Grype approach becomes advantageous when the SBOM itself serves as a standalone attestation artefact. Once generated, the SBOM can be stored, cryptographically signed, and shared with downstream consumers or auditors without needing repeated access to the original image. Separating the inventory (SBOM) from the vulnerability assessment permits independent verification and reuse across different tools and compliance frameworks.

Trivy's all-in-one model is preferable when operational simplicity and breadth are the priority in a CI pipeline. A single invocation covers vulnerabilities, secrets, container hardening checks, eliminating the need to coordinate multiple tools. This makes Trivy a practical choice for early feedback loops, despite the fact that it does not offer the same capability to persist a signed SBOM separately for later attestation.
